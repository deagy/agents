# Runner Adapters

This repository's agent definitions, routing, quality gates, and schemas are
plain text and data with no dependency on any model or agent runner. This file
is the one place that maps the abstract orchestration actions those definitions
assume onto the concrete mechanism of each supported runner. Skills reference
this file instead of restating runner-specific instructions inline.

Supported runners today: **Codex CLI** and **Claude Code**. A runner not listed
here still works manually: read the referenced `AGENT.md`, shared policies, and
task brief, and act as that role in a plain chat session; treat any "ask the
human" step as a normal turn back to the user.

| Action | Codex CLI | Claude Code |
|---|---|---|
| Invoke a skill | `$skill-name <args>` inline | `/skill-name <args>`, or the `Skill` tool |
| Ask the human a question | Prompt inline in the current turn | The `AskUserQuestion` tool, **main conversation only** |
| Spawn a subagent with an injected role prompt | Codex's subagent/task dispatch mechanism | The `Agent` tool |
| Install a plugin from a marketplace | `codex plugin marketplace add <path>` then `codex plugin add <plugin>@<marketplace>` | `/plugin marketplace add <path>` then `/plugin install <plugin>@<marketplace>` |

## The ask-the-human rule

**Only the top-level orchestrator turn asks the human.** A dispatched subagent
that reaches a decision only a human can make must stop and return a clearly
labeled blocking question in its result — it must not attempt to prompt the
human directly.

This is a hard constraint in Claude Code: a subagent invoked through the `Agent`
tool cannot call `AskUserQuestion`; only the main conversation can. It is kept
as the universal rule across every runner, including ones where a subagent
could technically interject, because centralizing human interaction in the
orchestrator is what keeps a multi-agent run legible: the human sees one
consolidated set of questions per round instead of being interrupted by
whichever subagent happens to hit a blocker first.

When a wave of dispatched subagents returns, the orchestrator collects every
blocking question they raised and asks them together in one turn rather than
resolving them one at a time.

## Plugin/skill packaging per runner

Both runners read the same skill and role content; only the wrapper differs:

- **Codex CLI**: `plugins/agentic-sdlc/.codex-plugin/plugin.json` declares the
  plugin; each skill under `plugins/agentic-sdlc/skills/<name>/` ships an
  `agents/openai.yaml` sidecar (interface/policy metadata Codex reads).
- **Claude Code**: `plugins/agentic-sdlc/.claude-plugin/plugin.json` declares
  the same plugin, pointing at the same `skills/` directory Codex uses — no
  skill content is duplicated. Claude Code needs no sidecar file per skill;
  its equivalent of Codex's `policy.allow_implicit_invocation: true` is simply
  the *absence* of `disable-model-invocation: true` in a `SKILL.md`'s
  frontmatter, which is already the default these skills use.

Subagent wrappers generated by `agentic_sdlc.py init` follow the same split:
`.codex/agents/<id>.toml` for Codex, `.claude/agents/<id>.md` for Claude Code,
generated from the same profile/catalog data (see `init --runner`).

## System-wide install

Both runners support installing a plugin at **global/user scope** instead of
per-project, which activates its bundled skills (and, for Claude Code, its
bundled subagents) in *every* project directory, not just the one that hosts
the marketplace:

- **Codex CLI**: plugin installs are user-scoped by default
  (`~/.codex/config.toml`); `codex plugin add <plugin>@<marketplace>` from
  within this repository is enough.
- **Claude Code**: pass `--scope user` explicitly —
  `claude plugin install <plugin>@<marketplace> --scope user` (or choose "user"
  scope when `/plugin install ...` prompts for it).

`plugins/secure-cloud-agents/` uses this to make this repository's own 34
specialist roles and 6 skills reachable from any project on the machine — see
that plugin's own `README.md` for why its pointer files are the one place in
this repository that hardcode an absolute, machine-specific path (a plugin is
cached on install, and cached content loses access to anything outside its own
directory, including the sibling `agents/` tree this repository's roles live
in — see [Path traversal
limitations](https://code.claude.com/docs/en/plugins-reference.md#path-traversal-limitations)).
`plugins/agentic-sdlc/` doesn't need this: it's designed to be copied into many
different target repositories, so its skills only ever reference paths inside
its own plugin directory.

Codex CLI has no plugin-bundled-subagent mechanism at all — custom agents are
only discovered from `.codex/agents/` (project) or `~/.codex/agents/` (global)
on disk. `plugins/secure-cloud-agents/codex-agents/*.toml` is therefore a
repo-tracked staging copy, not something Codex loads directly from the plugin;
copying (or symlinking) those files into `~/.codex/agents/` is a separate,
explicit step a human runs once, documented in that plugin's `README.md`.

## Project-local overrides

Global-by-default doesn't mean a project can't have its own version of a role.
The two runners handle this differently, and the difference matters for how an
override actually gets enforced:

- **Claude Code**: plugin-installed agents/skills are namespaced by the
  runner itself (`secure-cloud-agents:code-reviewer`), per the plugin
  reference's "cannot collide with other levels." A project's own bare
  `.claude/agents/code-reviewer.md` is a *different identifier*, not an
  override — both remain simultaneously invocable. There is no runner-level
  shadowing to rely on.
- **Codex CLI**: project-scoped (`.codex/agents/`) and global-scoped
  (`~/.codex/agents/`) custom agents share the same bare name, so a
  project-local `.codex/agents/code-reviewer.toml` is a genuine same-name
  collision with the copy under `~/.codex/agents/code-reviewer.toml` — Codex's
  own project-before-global resolution decides the winner.

Because Claude Code gives no free shadowing to lean on, `run-agent-orchestration`
(`.agents/skills/run-agent-orchestration/SKILL.md`, "Dispatch in Waves")
enforces the override itself: before dispatching a role, it checks for a
project-local `.claude/agents/<role-id>.md` or `.codex/agents/<role-id>.toml`
and dispatches that in preference to the global plugin version when one
exists. The knowledge store takes the same global-default,
project-local-override shape through a real code-level mechanism instead — see
`agents/knowledge-store/README.md`.
